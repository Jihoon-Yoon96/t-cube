# AGENTS.md

## 프로젝트 개요

- 이 프로젝트는 Nuxt 3, Vue 3, Pinia, pnpm을 사용하는 프론트엔드 애플리케이션이다.
- 작업 전에는 기존 폴더 구조, 컴포넌트 작성 방식, 상태 관리 방식, 서비스 계층 사용 방식을 먼저 확인한다.

## 기본 작업 원칙

- 요청받은 범위에 집중하고, 관련 없는 리팩터링이나 스타일 변경은 하지 않는다.
- 사용자가 이미 수정한 내용은 임의로 되돌리지 않는다.
- 변경 전후의 의도를 명확히 설명할 수 있는 작고 안전한 단위로 수정한다.
- 코드보다 설정, 문서, 데이터 변경이 더 적절한 문제라면 먼저 그 방향을 고려한다.
- 불확실한 요구사항은 코드베이스에서 근거를 찾아 판단하고, 위험한 가정이 필요할 때만 사용자에게 질문한다.

## 명령어와 패키지 관리

- 패키지 매니저는 `pnpm`을 사용한다.
- 의존성 설치는 `pnpm install`을 사용한다.
- 개발 서버는 `pnpm dev` 또는 환경별로 필요한 경우 `pnpm dev:local`을 사용한다.
- 프로덕션 빌드는 `pnpm build`로 확인한다.
- stage 빌드는 `pnpm build:stage`를 사용한다.
- `npm` 또는 `yarn` 명령은 사용하지 않는다.
- 새 production dependency는 꼭 필요한 경우에만 추가하고, 추가 이유와 대안을 설명한다.

## 코드 작성 규칙

- Vue/Nuxt 코드는 이 프로젝트의 기존 작성 방식을 우선한다.
- 페이지 진입점은 `pages`, 공통 레이아웃은 `layouts`, 화면 단위 UI는 `components`, 전역에 가까운 상태는 `stores`, 순수 처리 로직과 외부 연동은 `services`, 공통 타입은 `types`에 둔다.
- 컴포넌트는 기존처럼 Vue SFC 구조를 사용하고, 상태 변경은 가능한 한 Pinia store나 composable 역할에 맞춰 분리한다.
- TypeScript 친화적인 코드를 작성하고, 불필요한 `any` 사용은 피한다.
- 컴포넌트, 페이지, 레이아웃, 스토어, 서비스, 타입은 기존 디렉터리 역할에 맞춰 배치한다.
- API 호출, 비즈니스 로직, 상태 관리는 UI 컴포넌트 안에 과도하게 섞지 않는다.
- 주석은 기본적으로 "무엇을 하는지"를 설명한다.
- 변수명, 함수명, 타입만으로 이해하기 어려운 구현에는 "왜 이렇게 해야 하는지"도 함께 설명한다.
- 함수, `computed`, `watch` 훅, 라이프사이클 훅에는 반드시 주석을 작성한다.
- 주석은 JSDoc 형태를 유지한다.
- 함수 주석에는 가능한 한 파라미터와 리턴값 설명을 포함한다.
- 주석 어투는 문장형 종결 대신 명사형 또는 짧은 설명형을 사용한다. 예: "`값을 변경합니다.`" 대신 "`값 변경`".

## UI 작업 규칙

- UI는 템플릿 빌더 도구 화면처럼 작업 중심으로 구성한다.
- 현재 디자인 톤은 어두운 navy 계열 배경, indigo/violet 계열 강조색, 밝은 텍스트, 얇은 구분선, 좌측 사이드바 기반 레이아웃이다.
- 버튼과 메뉴는 기존처럼 아이콘과 짧은 텍스트를 함께 사용하고, 아이콘은 `TcubeIcon`과 Remix Icon 이름을 우선 사용한다.
- 화면 구성은 큰 홍보용 hero보다 작업 단계, 업로드, 편집, 미리보기, 속성 패널처럼 실제 워크플로우를 바로 수행할 수 있는 구조를 우선한다.
- 카드, 패널, 버튼은 기존처럼 적당한 radius와 명확한 hover/active/disabled 상태를 유지한다.
- 화면은 모바일과 데스크톱 모두에서 깨지지 않도록 반응형으로 작성한다.
- 텍스트가 버튼, 카드, 패널 영역을 넘치거나 다른 요소와 겹치지 않도록 확인한다.
- 단순한 장식보다 실제 사용 흐름, 가독성, 조작 편의성을 우선한다.
- 아이콘이 필요한 버튼이나 도구성 UI는 가능한 한 프로젝트에서 사용하는 아이콘 시스템을 따른다.

## 파일과 설정

- `.env` 값은 명시적인 요청 없이 수정하지 않는다.
- 빌드 산출물, 캐시, 로그 파일은 필요한 경우가 아니면 수정하거나 커밋 대상으로 만들지 않는다.
- `node_modules`, `.nuxt`, `.output` 내부 파일은 직접 수정하지 않는다.
- 설정 파일을 바꿀 때는 해당 변경이 런타임, 빌드, 배포에 미치는 영향을 함께 확인한다.

## 검증

- 코드 변경 후에는 변경 범위에 맞는 검증을 수행한다.
- 앱 전반에 영향을 줄 수 있는 변경은 가능하면 `pnpm build`로 확인한다.
- 실행하지 못한 검증이 있다면 이유를 명확히 남긴다.
- 테스트 스크립트가 추가되면 관련 변경에 대해 해당 테스트를 우선 실행한다.

## Git 안전 규칙

- 사용자가 명시적으로 요청하지 않는 한 destructive git 명령을 실행하지 않는다.
- 관련 없는 변경 파일은 건드리지 않는다.
- 커밋, 브랜치 생성, push는 사용자가 요청한 경우에만 수행한다.

## 응답 방식

- 변경한 내용, 확인한 내용, 남은 위험 요소를 간결하게 공유한다.
- 파일 경로나 명령어는 정확한 이름으로 표기한다.
- 불확실한 내용은 추측처럼 말하지 않고, 확인된 사실과 가정을 구분한다.
